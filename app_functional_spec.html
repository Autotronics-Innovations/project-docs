Screen 1

My Device

The primary home screen. Displays real-time device status, air quality context, protection metrics, mode controls, battery, and a stats preview. All live data is sourced from BLE events and an external AQ API.

BLE Raw Model: Every time mode changes, a new BLE event arrives with timestamp, mode ("normal" | "turbo" | "off"), and duration_seconds. Sessions are tracked server-side.

Battery Estimation

Due to device IC memory and timing constraints, battery % is not calculated in firmware. The app handles all estimation using the manufacturer's 0.5C discharge curve. Firmware sends raw voltage only.

Data sent by firmware
Sent on mode change, or every 5 min. If extremely low power: every 10–15 min.
FieldTypeRequiredDescription
battery_voltage_mvInt (mV)MustRaw battery voltage in millivolts. e.g. 3810 = 3.810V
charging_statusEnum / BoolMust"charging" / "not_charging" (or true/false)
sample_timestampDateTimeRecommendedDetermines freshness, supports filtering
measurement_validBoolOptionalFirmware flag indicating reading validity
Architecture decision
LayerResponsibility
FirmwareSend raw battery voltage + charging status only
App / FrontendAll battery state logic, filtering, interpolation, user display
BackendOptional — store voltage history, support analytics and tuning
Backend is not required for live battery display. App works standalone via BLE.

Battery State Model

The app must not always force a battery percentage. Battery is modelled as one of four states:

State A
⚡ Charging
Condition: charging_status = charging

UI: Show "Charging". Do not calculate % from live voltage. Optionally show last stable % captured before charging.
State B — Settling_after_Charging
⏱ Settling After Charging
Condition: Transition from charging → not_charging

UI: Show last stable % or "Battery updating". Do not use fresh voltage. Settling window: 5 minutes (tunable).
State C — Stable_after_Charging
✅ Stable / Discharging
Condition: not_charging + settling complete + enough valid samples

UI: Calculate and display user-facing battery %.
State D — Unknown
❓ Unknown
Condition: No valid reading / too few samples / stale data / invalid voltage / just connected

UI: Show "Battery unavailable" or last known stable value.

Calculation Logic — 8 Steps

1
Collect voltage samples
App receives raw battery_voltage_mv values from firmware over BLE.
2
Validate samples
Ignore: missing, corrupt, physically impossible, or firmware-flagged invalid readings.
3
Filter voltage
Average the last 5–10 valid samples. This filtered value is the input to estimation. Optionally: store at 2, 2, 1 min → average → show at 5th min. Else send raw.
filtered_voltage = average(last_5_to_10_valid_samples)
4
Only estimate during stable/discharging state
All three conditions must be true: not charging + settling complete + enough valid samples.
5
Convert voltage → discharge % via lookup table
Use the manufacturer's 0.5C discharge curve. Find two nearest voltage points and linearly interpolate.
Reference Curve (0.5C)
Voltage (V)Battery %Voltage (V)Battery %
4.16100%3.8038%
4.1095%3.7630%
4.0485%3.7222%
3.9875%3.7018%
3.9265%3.6814%
3.8855%3.6610%
3.8445%3.640%
6
Interpolation formula
Battery % = Plow + ((V - Vlow) / (Vhigh - Vlow)) × (Phigh - Plow)

// Where Vhigh >= V >= Vlow (two nearest lookup table points)
// & Plow → percentage at Vlow & Phigh is percentage at Vhigh
7
Clamp output
battery_pct = clamp(estimated_pct, min=0, max=100)
8
Stabilize UI output
Do not update on tiny fluctuations. Only update when % changes by at least 1–2%. Avoid sudden upward jumps unless device was charging.
Charging-specific rules
RuleBehaviour
While chargingNever calculate battery % from live voltage
Charging beginsKeep last known stable %, switch UI to charging state
Charging endsStart settling timer; do not estimate until settled
After settlingResume normal filtering and estimation

Edge Cases

CaseRiskHandling
Just unplugged from chargerVoltage remains artificially highEnter settling state for 5 min before recalculating
Temporary load spikeVoltage dips briefly → false battery dropUse filtered voltage, not single raw readings
App just connectedNot enough samples for reliable estimateDo not calculate until minimum valid sample count is reached
Invalid voltageCorrupt reading → nonsense %Reject out-of-range values; do not update UI

Stats & Insights

Weekly analytics dashboard. All metrics are computed server-side by joining BLE session data with AQ API data and user behaviour patterns. The BLE device tells you when and how hard it ran — the app constructs all meaning.

e.g. 82 · +12 vs last week · Pie chart fill

  coverage_score:    (protected_hours / total_awake_hours) × 40,
  smart_usage_score: (turbo_during_high_aqi / total_turbo_hours) × 30,
  streak_score:      (days_worn_this_week / 7) × 30
)  // Output: 0–100

delta_vs_last_week = this_week_score - last_week_score
// Green if positive, red if negative
pie_fill = score / 100

Data Layer

BLE Raw Data Model

BLE_EVENT { timestamp: DateTime, mode: "normal" | "turbo" | "off", duration_seconds: Int // how long current mode has been running } // Every time mode changes, a new event arrives. // Sessions are tracked server-side.

Backend Database Schema

sessions_table: user_id, device_id, start_time, end_time, mode, duration_seconds daily_aggregates: user_id, date, total_on_seconds, turbo_seconds, normal_seconds, smart_seconds, protection_pct, aqi_avg, avoided_events weekly_aggregates: user_id, week_start, weekly_score, streak_days, turbo_pct, best_day, total_protected_hours, total_avoided_events insights: user_id, generated_at, insight_type, message, priority achievements: user_id, achievement_id, unlocked_at
